Package management 🔗

apt
Advanced Package Tool

Software on GNU+Linux systems is mostly managed through package managers. Package managers provide access to a large repository of software packages. Elevated permissions are required to install or configure packages; generally only the root user can install or uninstall software.

APT

Package repository configuration for systems using apt is done via the directories in /etc/apt, mostly by adding files to /etc/apt/sources.list.d/ to add and enable additional repositories.

Repositories (on Ubuntu) are categorized into the following components:

  • main: Officially supported software
  • restricted: Supported software that is not available under a completely free license
  • universe: Community maintained software
  • multiverse: Software that is not free

Additionally, updates are split across the following repositories:

  • security: Important security updates
  • updates: Recommended updates
  • backports: Unsupported updates (updates from newer versions of Ubuntu)
  • proposed: Pre-release updates (will eventually be moved to updates)
Examples
apt update && apt upgrade
Update catalog and install upgrades
apt install nginx
Install the nginx webserver
apt remove firefox
Uninstall the Firefox browser

Snap

Canonical (the company behind Ubuntu) provides an additional tool to to install self-contained packages, called snap. Snap can be (and is) used alongside .deb packages installed via apt.

A single snap package might be provided through different channels, which allows users to easily switch to a different version of the same software (say to an extended support release or to a beta version).

By default, snaps are sandboxed and have limited access to the system. Certain packages require full system access and need to installed with classic confinement.

Examples
snap install firefox
Install Firefox
snap refresh firefox --channel=latest/beta
Switch Firefox snap to the beta version
snap remove firefox
Remove the Firefox snap
snap install code --classic
Install VS Code with classic confinement (full system access!)

Docker builds

Docker is a way to containerize applications, but also to provide a specific environment when building software.

Each instruction in a Dockerfile (e.g. RUN) creates a new layer. Consequently, once a file is part of a layer, removing it later in a different layer will not reduce the image size. Thus, when installing packages in a Dockerfile, make sure to remove any temporary files and caches within the same instruction:

RUN apt update \
	&& apt install -y package-to-install \
	&& rm -rf /var/cache/apt/archives/ /var/lib/apt/lists/*

Docker will also cache layers and only rebuild the layers that have changed. This means that it pays off to think about order of instructions, when writing a Dockerfile: keep instructions that change infrequently to the front and move the dynamic parts to the back.

When building Docker images for later distribution, you usually want it to only contain the build artifacts and runtime dependencies required to run your software. You don't want to ship the compiler infrastructure or the source code of your software. That's where multistage builds come in.

By providing multiple FROM instructions in a single Dockerfile, a multistage build is performed. Later stages can copy files from earlier stages, but only the last stage will be part of the final image.

FROM eclipse-temurin:25-jdk AS builder
WORKDIR /opt/app
COPY mvnw pom.xml src/ ./
RUN ./mvnw clean install

FROM eclipse-temurin:25-jre
WORKDIR /opt/app
EXPOSE 8080
COPY --from=builder /opt/app/target/app.jar /opt/app/app.jar
ENTRYPOINT ["java", "-jar", "/opt/app/app.jar"]

Networking

ping
Send ICMP ECHO_REQUEST to network hosts
mtr
A network diagnostic tool combining traceroute and ping
dig
DNS lookup utility
nc
netcat, perform arbitrary network requests
ss
dump socket statistics

Inevitably, you will run into network and connectivity issues sooner or later. Several tools can be used to perform common troubleshooting tasks.

ping sends ICMP messages to a given host and waits for the host to respond. Since you can't assume the DNS to be working, it is often a good idea to ping well-known IP addresses, such as 8.8.8.8 (Google DNS) or 1.1.1.1 (Cloudflare DNS).

mtr combines ping and traceroute: it pings all hosts on the route to a given host and records response times and packet loss rates for each hop. Pressing d cycles trough a textual and two visual display modes.

To check DNS (mis)configuration, use the tool dig or nslookup. To resolve the IP address of the host example.com, use dig example.com +short or dig @8.8.8.8 example.com +short to execute the DNS query against a specific or custom DNS server.

If an application does not start because the port is already in use, ss can shed light. Most commonly, it is called with the five option flags -plunt (show ports, show listening sockets, show UDP sockets, do not resolve service names, show TCP sockets) to show both TCP and UDP connections and listening sockets.

If your application cannot connect to a remote service, such as a database, but the remote host responds to ping and DNS is working, it can be helpful to connect to the service's port directly. This can be achived with nc or telnet, which can open a connection to arbitrary ports on remote hosts.

Examples
ping 8.8.8.8
Ping Google's DNS server
mtr 8.8.8.8
Measure response times and packet loss at each hop to the destination
dig example.com +short
Resolve IP address of host example.com
dig @1.1.1.1 example.com +short
Resolve IP address of host example.com, using Cloudflare's DNS server
ss -plunt
Show used ports of the local system
printf 'GET / HTTP/1.1\r\nhost:example.com\r\n\r\n' | nc example.com 80
Send a raw HTTP request to example.com

Find

find
Find files

Use find to find files in your local file system and filter them with conditions or execute actions on them

All conditions must match for a file to be printed, unless combining them with -o. Group conditions with '(' and ')' and negate a condition with '!'. The following conditions are commonly used; for a full list consult the man page.

-name
Filter by name (case-sensitive)
-iname
Filter by name (case-insensitive)
-size
Filter by size
-type
Filter by type (f=file, d=directory)
-empty
Filter empty files or directories
-writable
Filter writable files or directories
-readable
Filter readable files or directories
-executable
Filter executable files or directories
-mtime N
Filter for files or directory whose data was modified N*24 hours ago
-ctime N
Filter for files or directory whose status was modified N*24 hours ago
-mmin N, -cmin N
Filter for files or directory whose date/status was modified N minutes ago

Numeric arguments for conditions can be prefixed with + or - to check for greater-than or less-than, respectively.

The default action for a matched file is to print its path, but it's possible to define custom actions to perform on the matches:

-delete
Delete matched files (without confirmation!)
-exec command {} ';'
Execute the specified command for each matched file. Make sure to escape or quote the terminating semicolon.
-exec command {} +
Similar to above, but pass multiple files to the specified command at once.
Examples
find /tmp -type f
Find all regular files in directory /tmp
find -type d -empty
Find empty directories
find -type f -ctime +365 -delete
Delete files older than 1 year
find -size +1000M
Find files larger than 1000 megabytes
find -cmin -5
Find files modified in the last 5 minutes
find -iname '*.txt' -exec cat {} ';'
Find all text files and print their content (calling cat once per file)
find '(' -iname '*.txt' -o -iname '*.md' ')' -exec cat {} +
Find all text or markdown files and print their content (calling cat only as often as required)

SSH (Secure Shell)

SSH allows to connect to remote host securely over an encrypted connection. By using public and private keys, you can authenticate with a remote server without sharing any passwords.

Create a key pair and copy it to a remote server with two simple commands. It's generally safe to generate keys with the default options, but can easily be customized. The public part of a key pair is stored in a file with extension .pub (e.g. ~/.ssh/id_ed25519.pub).

When copying your public key to a server, you will be prompted for your password. It's also possible to manually copy the public key to the server, e.g. by sending it via email to the server admin. The public keys which allow authenticating with a server account are stored in ~/.ssh/authorized_keys (one key per line)

ssh-keygen
ssh-copy-id server.name.or.ip.address

Stating the obvious: Keep your private keys private and do not share them!

SSH connections also allow forwarding local or remote ports, tunneling the data through the secure connection. Use ssh -L1337:example.com:80 to forward the local port 1337 to port 80 of host example.com, and ssh -R1337:example.com:80 to forward the remote port 1337 to port 80 of host example.com. This can be especially useful when forwarding connections to the local client or to the SSH server itself, by specifying localhost as the target address.

SSH keys are only available on the local host, but they can be automatically be made available on remote hosts by specifying the -A option when connecting to the server or by adding ForwardAgent yes to the host connection configuration in ~/.ssh/config.

Not only does SSH allow interactive remote sessions, but the protocol allows copying files to and from servers too. scp (secure copy) is the command to use here. Commonly used options are -r to copy directories recursively (including their content) and -C to compress files before sending.

While scp suffices for the simple use cases, rsync offers more options and flexibility, such as only transferring changed files or making sure remote directories match the local directories exactly. Common options are -a (archive), -u (update), --delete, and -z (compress). rsync can also be used to synchronize two local directories and does not necessarily need to involve a remote server.

To quickly see running processes on a (local or remote) system, run top (interactive, ? to bring up help) or ps aux (snapshot).

If a process gets stuck or does not respond, and needs to be terminated, you can use the command kill to send one of several signals to the process. Run kill -l to get a list of available signals on your system. Without specifying the signal, 15 (SIGTERM) will be the default. If that still didn't terminate the process, you can use the (uncatchable) signal 9 (SIGKILL), which forcefully stops and removes the process.

GNU+Linux is generally case-sensitive. Consequently, command names and file names must be entered exactly as shown; readme.TXT and README.txt are two distinct files.

Prompt

Usually, shells display a prompt string (PS1) when waiting for users to enter their command line. The prompt string can be as simple as a single dollar sign, but generally it is configured to include useful information such as the logged in user, host name, and working directory.

For regular users, the prompt string ends with $; and with # for the root user (the system administrator):

username@hostname:/path/to/working/directory$ _

Command lines are submitted by pressing the return/enter key ↲. Upon submission, the shell parses the command line and executes the command(s).

Online tutorials or sample commands usually prefix lines to be entered into the CLI with $ only. This document follows that convention.

Command Structure

Most commands follow the same pattern: the first word in the command line specifies the binary (program) to execute, optionally followed by parameters. Parameters can be program options or paths to files or directories. Options usually start with a single or two dashes.

$ program -o --option file1 /path/to/another/file

Binaries are located in the directories from the environment variable PATH, unless the command name contains a slash.

Comments can be introduced by prepending text with a hash (#). Everything after the hash is ignored by the shell. This is especially useful in scripts:

$ echo 'Hello RISC!' # this is a comment
Hello RISC!
$ # When is next Monday? (print email format)
$ date -d 'next monday' --rfc-email
Mon, 29 Sep 2025 00:00:00 +0200

Short options (with a single dash) can be combined for most binaries: program -x -y -z is equivalent to program -xyz; but ls --all is different from ls -all, which is equivalent to ls -a -l -l.

All commands and their options are case-sensitive, for instance ls -a is different from ls -A.

Every command comes with a user manual. Use the program man to display and read those system reference manuals:

$ man man # display the manual for the program "man" (the manual pager)
$ man echo # display the manual for the program "echo"

The Synopsis section shows how a command must be called and which combinations of parameters are valid.

When viewing the User manuals (often called man pages), the keyboard can be used to scroll and search:

q
Quit (return to the shell)
/
Forward search
?
Backward search
n
Jump to next match
N
Jump to previous match

In the QWERTY keyboard layout, both / and ? share the same physical key.

To get help about the shell's builtins, run help, e.g. help cd.

Filesystem Navigation 🔗

cd
change directory
pwd
print working directory
ls
list directory contents
cat
concatenate files and print on standard output
less
show long text in a pager

Directories are organized hierarchically in a tree structure. The root directory in this hierarchy is designated as /. Every other file and directory of a system is nested in this directory or its subdirectories. File and directory names are case-sensitive.

Paths can be categorized into two types:

Absolute
Always start with a slash (/). They are valid independent of the current working directory and always describe the same file. Think of them as "addresses". (paths starting with ~ are expanded by the shell to start with the home directory of the user).
Relative
All other paths (not starting with a slash). They depend on the current working directory. Have them start with ./ to explicitly show that they are relative to the current directory (.). Think of them as "signposts" guiding your way to to the target

Hidden files (and directories) have names starting with a period (.), e.g. .profile.

To get the absolute path of the current directory, one can use the pwd command to print the working directory. ls will list the content of a directory (current directory by default) and cd allows changing directories.

To view the content of a (text) file, use the cat command to output the raw content of the file to the terminal. For longer files, less is recommended, which allows paging through the file. It uses the same shortcuts as man – in fact, man pages are actually displayed by the less binary.

Notable Files and Paths

.
The current directory
..
The parent directory
/dev/null
A special device file which discards any data written to it
/dev/random
A special device file which produces random bytes

man hier contains a list of commonly available directories on a GNU+Linux system and their purpoes, as defined by the FHS.

Working with Files and Directories 🔗

touch
create empty files or update modification time of existing files
mkdir
make directory
rm
remove files
rmdir
remove empty directories
cp
copy files and directories
mv
move (or rename) files and directories
ln
create links to files and directories

Use touch and mkdir to create empty files and directories, respectively. Both commands support creating multiple files at once. To create nested directory hierarchies, mkdir -p is quite useful.

To remove files again, use the rm command. rmdir removes empty directories. To remove non-empty directories including their files and subdirectories, rm -r can be used.

There is no confirmation when deleting or overwriting files!

Deletion is final. There is no trash or recycle bins to recover files from!

Both cp and mv commands support two forms: a two-parameter form and a 3+ parameter form. mv src dst moves src to dst, whereas mv src1 src2 srcN dst moves all src… files into the dst directory.

To copy directories including their files and subdirectories, call cp with the -R or -r option.

Wildcards

It is often useful to specify groups of files. Shells offer this possibility via “wildcards” or “globs”.

?
exactly one character
*
arbitrary many characters (even no characters)
[xyz]
exactly one character: either x, y, or z
[0-9]
exactly one digit
[a-f]
exactly one character between a and f, inclusive
[0-9a-fA-F]
exactly one hexadecimal digit
[!…]
exactly one character not in the set, i.e. the complement set

Wildcards are expanded by the shell to a list of matching paths. Binaries never see the wildcard metacharacters, but only the expanded list. If a glob character needs to be passed to a binary verbatim, it must be escaped with a backslash (\) or surrounded by quotes ('). Escaping and quoting protect the glob pattern from being expanded.

Wildcards only match within a directory; they never match the slash directory separator (/).

The leading period of hidden files is not matched by wildcards either. If hidden files need to be matched, the period must be specified explicitly in the pattern.

Examples
*.pdf
All (non-hidden) PDF files in the current directory
_*
All file names that start with an underscore
*risc*
All (non-hidden) files that contain risc in their name
*
All non-hidden files in the current directory
* .*
All files in the current directory, including hidden files
*/
All non-hidden directories
/home/*[0-9]/
All home directories ending in a digit
????.txt
All non-hidden text files with exactly 8 characters in length (4 + 1 + 3)
[!aeiou]*
All non-hidden files that do not start with a lower-case vowel

Some shells implement additional expansion patterns. One such shell is bash which implements Brace Expansion. With brace expansion {x,y}{19..21} expands into x19 x20 x21 y19 y20 y21, independent of any existing files.

Text Processing 🔗

wc
word count
head
Show first part of input
tail
Show last part of input
cut
Print selected columns of files

GNU+Linux comes with many powerful tools to transform and filter text-based data. Most programs operate on lines, reading the input line by line and printing the processed output. Input files are generally not modified.

Counting Lines and Words

wc counts the number of lines, words, and bytes.

wc -l prints the line count only, wc -w prints the word count, and wc -c prints the byte count.

$ cat file
this file contains
three lines and
just nine words.

$ wc file
 3  9 52 file

$ wc -l file
3 file

$ wc -w file
9 file

$ wc -c file
52 file

Start and End of Text

head and tail print the first or last part of an input, respectively. The default is to show 10 lines, which can be changed with the -n option.

head -n3 or head -3 will only show the first 3 lines. Negating the number will print everything except the last N lines: head -n-3 will exclude the last 3 lines of the file.

tail -n3 or tail -3 will only show the last 3 lines. tail -n+3 will skip the first 2 lines (printing line 3 until the end).

$ cat file
one
two
three
four
five

$ head -2 file
one
two

$ head -n-2 file
one
two
three

$ tail -n2 file
four
five

$ tail -n+2 file
two
three
four
five

Extracting Fields

cut extracts fields or characters from a text. Select fields and fields ranges with -f or characters and character ranges with -c. Omitting one side of a range selects all fields/characters from the start or until the end of the line. The default field delimiter is the tab character, but it can be changed with -d. cut always outputs the fields in the same order as the input.

If you have a colon-delimited file and you want to print the first three and the seventh field, run cut -d: -f1-3,7. To extract the first ten characters of every line, run cut -c1-10.

$ cat file
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
bin:x:2:2:bin:/bin:/usr/sbin/nologin
sys:x:3:3:sys:/dev:/usr/sbin/nologin

$ cut -d: -f1,7 file
root:/bin/bash
daemon:/usr/sbin/nologin
bin:/usr/sbin/nologin
sys:/usr/sbin/nologin

$ cut -d: -f-4 file
root:x:0:0
daemon:x:1:1
bin:x:2:2
sys:x:3:3

$ cut -d: -f6- file
/root:/bin/bash
/usr/sbin:/usr/sbin/nologin
/bin:/usr/sbin/nologin
/dev:/usr/sbin/nologin

Sorting

Sort order depends on the active locale. If a specific sort order is required, you can override the locale by setting the LC_ALL variable: LC_ALL=C sort ….

Use sort to sort the lines of an input alphabetically. -r reverses the sort direction.

Switch to numerical sorting with the -n option; non-numeric input is interpreted as the value 0.

For multi-field inputs, specify the field separator with -t (default: non-blank to blank transition). Specify the sort keys with -k.. Sort keys are specified as KEYDEF, which specifies the start and stop position plus any options. If the stop position is missing, the rest of the line is used!

The default sort algorithm is not stable! If stable sorting is required, pass -s.

--debug prints detailed information about the keys used and the sort process.

Removing Duplicates

uniq removes repeated lines from a text. Repeated lines are only removed if they are adjacent.

uniq -c counts repeated lines, prepending each output line with the number of repetitions. This format allows reconstructing the original text. It is also useful for quick frequency analyses of inputs.

uniq -d prints only lines with repetitions and uniq -u prints only lines without.

$ cat file
a
r
r
a
y

$ cat sorted-file
a
a
r
r
y

$ uniq file
a
r
a
y

$ uniq sorted-file
a
r
y

$ uniq -c file
      1 a
      2 r
      1 a
      1 y

$ uniq -c sorted-file
      2 a
      2 r
      1 y

$ uniq -d sorted-file
a
r

$ uniq -u sorted-file
y

I/O Redirection 🔗

Input and output of commands can be redirected from and to files, or passed to a second process. The redirection is set up by the shell when starting the process. Programs do not need to implement anything themselves; redirection happens transparently.

Being able to combine input and output of arbitrary commands unlocks the true potential of the command line!

< file
<0 file
Read file as (standard) input (STDIN).
> file
1> file
(Standard) output (STDOUT) of the command is redirected into the file. Existing files are truncated.
>> file
1>> file
(Standard) output of the command is appended to the file.
2> file
(Standard) error (STDERR) of the command is redirected into the file. Existing files are truncated.
2>> file
(Standard) error is appended to the file.
2>&1
Duplicate standard error from current standard output. Order of redirection operators matter, to redirect both stdout and stderr into a single file, use > file 2>&1
|
Pipe the output of the left command to the input of the right command. Multiple pipes and commands can be chained.
$(cmd)
Command substitution: The output of the command will be inserted into the command line in place of the parenthised expression. Alternatively, backticks can be used: `cmd`.

There is no confirmation when overwriting existing files!

Examples
ls | less
Scroll a long directory listing
grep 'a' file | sort | uniq -c | sort -n
Count frequency of all lines containing the letter 'a'
find 2> /dev/null
Find files and discard error messages
ping localhost >> pings
Append output of ping to a file
echo "You executed $(wc -l < ~/.bash_history) commands"
Print a message about your shell usage

Text Search 🔗

grep
grep (global regex print)

GNU+Linux comes with programs to perform efficient and effective text search. The most popular is probably grep. Grep leverages Regular Expressions (a.k.a. re, regex, regexp) to find patterns in files or text. Every line that matches the pattern is printed; lines not matching the pattern are omitted from the output.

There are different flavors/implementations of Regular Expressions and grep parses the search pattern as BRE by default. BREs only support a minimal set of meta characters. grep -E parses the pattern as ERE.

.
One arbitrary character (except line breaks)
[xyz]
[0-9]
[a-f0-9xyz]
One arbitrary character from the list or range
[^xyz]
One arbitrary character from the complement set of the list or range
^
Anchor the pattern at beginning of the line
$
Anchor the pattern at the end of the line
*
The preceding expression can be repeated or omitted
?
The preceding expression is optional ERE
+
The preceding expression is mandatory and can be repeated ERE
{N}
The preceding expression must be repeated N times ERE
{0,N}
{,N} GNU
The preceding expression may be repeated at most N times ERE
{N,}
The preceding expression must be repeated at least N times ERE
{M,N}
The preceding expression must be repeated at least M and at most N times ERE
()
Group subexpressions ERE
|
Specify alternatives ERE

Grep's behavior can be additionally controlled through options:

-i
Ignore case
-v
Invert: only print lines without match
-o
Only print matching part of lines
-c
Count number of matches per file
-n
Show line number of matches
-h
Hide file names
-l
Llist file names only
-q
Quiet mode (no output)
-r
Search directories recursively
-A N
Show N lines After each match GNU BSD
-B N
Show N lines Before each match GNU BSD
-C N
Show N Context lines around each match GNU BSD

Because some regex meta characters conflict with shell glob characters, the general recommendation is to quote the pattern of the grep command

Examples
grep ':/bin/sh$' /etc/passwd
Find users with /bin/sh as Login-Shell (the last field contains the login shell)
grep -E 'seriali[sz](ed?|ing|ations?)'
Find American and British spelling of words
grep -ic 'RISC'
Count occurrences of 'RISC' in files, ignoring case
grep -o '[-+]\?[0-9]*'
Extract numbers from a text
grep -e 'risc' -e 'RISC'
Find 'risc' and 'RISC' (but not 'Risc' or 'RiSC')
grep -B1 '^='
Find headings in a markdown document by including the line preceding the markup
grep -v '^$'
Omit empty lines: print non-empty lines only
grep -v '^ *#'
Print non-comment lines for languages such as Python

Permissions 🔗

id
Print user and group IDs

Display file status
chmod
change file mode bits
chown

sudo
execute a command as another user (substitute user do)

Permissions in GNU+Linux always apply to the file data itself. Not to the file name (!) – this becomes relevant when a file has multiple hard links.

Permissions can be applied to different users:

user
the user who owns the file.
group
users in the file's group.
others
other users not in the file's group (sometimes called “world”).

Each user can be assigned the following permissions:

read 4
  • File content can be read.
  • Directory content can be listed.
write 2
  • File content can be written.
  • Files can be created/deleted/moved inside a directory.
execute 1
  • Files can be executed.
  • Can change to directory and its subdirectories; can access files inside directory.

The numeric values allow representing all three permissions as a single octal digit: simply sum the values all active permissions. To designate a number as octal, prepend it with 0, e.g. 0644 for user=readwrite,group=read,others=read).

To display the permissions of files in both symbolic and octal notation, execute stat with the file(s) as parameter.

The additional permission "Set User-ID" (SUID bit) has the kernel execute a program always with the permission of its owner. This is required for certain tasks, such as changing your own password. The SUID bit is displayed as a lower-case s in place of the user's execute permission, e.g. rws-r-xr-x.

Permissions of a file are changed with the chmod command, which allows specifying the new permissions as octal number or in symbolic notation.

Only the owner of a file (and root) can change the permissions of a file.

The root user can change the owner and the group of a file with the chown command. Owner and group are separated with a colon (:).

Only root can change the owner and group of a file!

sudo is a mechanism to execute commands as a different user (most commonly as root), without having to share passwords. Unless specified otherwise via -u, the default user is root. Configuration of who can execute what is configured via the /etc/sudoers file, see man 5 sudoers.

User names and password digests are stored in the files /etc/passwd and /etc/shadow. (man 5 passwd, man 5 shadow)

Examples
stat file
Show the permissions of the file.
chmod u+x file
Make the file executable for its user.
chmod 0777 file
Enable all permissions for the file (not recommended!)
chmod u=rwx,go-w file
Enable all permissions for the user (owner) of the file, remove write permissions of group and others.
chown dfalkner:risc file
Change file owner to 'dfalker' and file group to 'risc'.
